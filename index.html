Cisco RADIUS Cisco-AVPair 1 General-purpose attribute used for various configurations (shell privilege level, ACL assignment, VLAN assignment, SGT assignment, QoS policy, Bandwidth control, etc.)
ACLs User Roles VLAN Assignment URL Redirects Captive Portals SGT DACL QoS Bandwidth CoA VPN
Both Cisco-AVPair = "shell:priv-lvl=15" for admin access, Cisco-AVPair = "ip:sub-qos-policy-in=QOS_POLICY" for QoS policy assignment
Implementation Steps
  1. Use 'Cisco-AVPair' to reference ACL in the format 'ip:inacl#<acl-name>'.
  2. Enable downloadable ACLs (dACLs) with 'aaa authorization network'.
  3. For VLAN overrides, use 'Tunnel-Private-Group-ID' and configure dynamic VLANs on interfaces.
  4. For SGT assignment, use 'cts:security-group-tag=xx' where xx is the tag number.
  5. For QoS policy, use 'ip:sub-qos-policy-in=QOS_POLICY' and 'ip:sub-qos-policy-out=QOS_POLICY'.
  6. For bandwidth control, use 'ip:sub-qos-policy-in=BANDWIDTH_POLICY' with rate limiting.
  7. For VPN configurations, use 'vpn:addr-pool=VPN_POOL_NAME'.
  8. For CoA triggers, configure 'aaa server radius dynamic-author'.

Fortinet Vendor-Specific Attributes

About Fortinet Authentication Attributes

Fortinet devices support a wide range of vendor-specific attributes (VSAs) for both RADIUS and TACACS+ authentication. These attributes allow for granular control over user access, including administrative privileges, VDOM restrictions, web filtering overrides, and more.

The Fortinet RADIUS vendor ID is 12356. For RADIUS VSAs to work properly, your RADIUS server must have the Fortinet dictionary installed to recognize these attributes.

Official Fortinet RADIUS VSA Documentation | Official Fortinet TACACS+ Documentation

Important Implementation Notes:

  • FortiGate devices require that the RADIUS or TACACS+ server be configured as an authentication source before VSAs can be used.
  • For TACACS+ administrator authentication with attribute overrides, you must enable accprofile-override and/or vdom-override in the system admin configuration.
  • Many VSAs are case-sensitive and must match exactly with the corresponding settings on the FortiGate.
  • For multi-VDOM environments, make sure to configure RADIUS or TACACS+ servers in the correct VDOM.

FortiGate Administrator Authentication Example

This example shows how to configure RADIUS and TACACS+ for FortiGate administrator authentication with proper attribute settings.

FortiGate Configuration

config system admin
    edit "radius-admin"
        set remote-auth enable
        set accprofile-override enable
        set vdom-override enable
        set remote-group "network-admins"
    next
end

config user radius
    edit "RadiusServer"
        set server "192.168.1.100"
        set secret "radiusSecret"
        set auth-type auto
    next
end

config user tacacs+
    edit "TacacsServer"
        set server "192.168.1.101"
        set key "tacacsSecret"
        set authen-type auto
        set authorization enable
    next
end

config system global
    set admin-radius-server "RadiusServer"
    set admin-tacacs+-server "TacacsServer"
end

RADIUS Server Configuration (Example: FreeRADIUS)

In the users file or equivalent user database:

admin_user   Cleartext-Password := "password"
            Fortinet-Access-Profile = "super_admin",
            Fortinet-Vdom-Name = "root"

region_admin Cleartext-Password := "password"
            Fortinet-Access-Profile = "prof_admin",
            Fortinet-Vdom-Name = "europe_vdom"

TACACS+ Server Configuration (Example: tac_plus)

user = admin_user {
    login = cleartext "password"
    service = shell {
        Fortinet-Access-Profile = "super_admin"
        Fortinet-Vdom-Name = "root"
    }
}

user = region_admin {
    login = cleartext "password"
    service = shell {
        Fortinet-Access-Profile = "prof_admin"
        Fortinet-Vdom-Name = "europe_vdom"
    }
}

FortiGate Content Filtering Example

This example shows how to use Fortinet RADIUS VSAs to customize web filtering and application control for specific users.

FortiGate Configuration

config user radius
    edit "ContentFilterRadiusServer"
        set server "192.168.1.100"
        set secret "radiusSecret"
        set auth-type auto
    next
end

config firewall policy
    edit 1
        set name "Internet_Access"
        set srcintf "internal"
        set dstintf "wan1"
        set srcaddr "all"
        set dstaddr "all"
        set action accept
        set schedule "always"
        set service "ALL"
        set utm-status enable
        set webfilter-profile "default"
        set application-list "default"
        set fsso enable
        set groups "RADIUS_Users"
    next
end

Cisco-AVPair Attribute

The Cisco-AVPair attribute (vendor-specific attribute 1) is extremely versatile and can be used for numerous configuration options. Here are some of the most common uses:

Purpose Syntax Example Description
Admin Privilege Level shell:priv-lvl=N shell:priv-lvl=15 Sets the administrative privilege level (0-15) for CLI access
ACL Assignment ip:inacl#N=ACL_ENTRY ip:inacl#10=permit ip 10.0.0.0 0.255.255.255 any Assigns an inbound ACL entry to the user session
Outbound ACL ip:outacl#N=ACL_ENTRY ip:outacl#20=deny ip any 192.168.0.0 0.0.255.255 Assigns an outbound ACL entry to the user session
VLAN Assignment cisco-avpair="vlan=VLAN_ID" cisco-avpair="vlan=100" Assigns the user to a specific VLAN (alternative to IETF Tunnel attributes)
SGT Assignment cts:security-group-tag=N cts:security-group-tag=3 Assigns a Security Group Tag to the user session
QoS Policy ip:sub-qos-policy-in=POLICY_NAME ip:sub-qos-policy-in=HIGH_PRIORITY Applies a QoS policy to inbound traffic
AnyConnect VPN Group webvpn:vpn-group-policy=GROUP_NAME webvpn:vpn-group-policy=SALES_VPN Assigns the user to a specific VPN group policy on ASA
